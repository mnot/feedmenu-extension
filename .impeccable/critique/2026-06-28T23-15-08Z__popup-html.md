## Anti-Patterns Verdict

**Does this look AI-generated? No.**

**LLM assessment:** The per-platform native skinning is a deliberate, non-generic decision — the opposite of the one-house-style reflex. Glass is purposeful (Safari-only), not decorative-by-default. No eyebrows, no gradient text, no icon+heading+text card grid, no hero-metric. The restraint reads as intent, not timidity.

**Deterministic scan:** `detect.mjs` on popup.html returned `[]` — zero findings. Clean.

**Visual overlays:** Not injected on the live target — the popup calls `chrome.*` and freezes at the loading state outside the extension runtime. Visual evidence instead came from this session's `dev/preview.html` harness, which drives the real `popup.js` through a stubbed `chrome`: all three skins × light/dark, plus loading/error/empty/expanded-submenu, were rendered and reviewed.

## Overall Impression

Genuinely well-built and on-brand: distinct, honest states; keyboard-reachable; AA contrast; three native skins. The single biggest gap is the mismatch between the **stated audience** (curious mainstream users who may not know what a feed is) and the **copy**, which quietly assumes they do. The visual craft is ahead of the verbal craft.

## What's Working

- **Native-per-platform skinning.** The defining move. It's distinctive, anti-slop, and serves the "feel native" principle directly.
- **Honest, distinct states.** Loading, found, empty, and error are now four clearly different things — the error glyph can't be mistaken for a stalled spinner.
- **Accessibility depth.** Keyboard-operable submenu (`aria-expanded`/`aria-controls`), visible focus rings, AA contrast (including the Safari secondary-text fix), `prefers-reduced-motion`, and modifier-click preserved for power users.

## Priority Issues

- **[P1] Jargon vs. the newcomer audience.** PRODUCT.md's primary user "may not know what a 'feed' even is," yet the UI leans on "feed" throughout and the empty state reads "Checked /.well-known/feed-menu.json" — developer-facing. The newcomer is told *what wasn't found* in terms they can't parse.
  - **Why it matters:** It breaks the core "self-explanatory by default" principle for exactly the person it's meant to serve.
  - **Fix:** Humanize the empty state (drop the well-known path or demote it to a tiny detail); consider a one-line "what's a feed?" explainer in the empty/first-run case.
  - **Suggested command:** /impeccable clarify

- **[P1] Error state has no recovery path.** Errors render clearly but offer no in-context "Try again," and "Server error: 500" exposes a raw code. The only retry is the header reload icon, not associated with the error.
  - **Why it matters:** A user who hits a transient failure has no obvious next action.
  - **Fix:** Add a "Try again" button inside the error block; rewrite messages to plain language with a suggestion ("Couldn't reach the site — check your connection and try again").
  - **Suggested command:** /impeccable harden

- **[P2] Stale header title on reload-into-error.** After a successful load sets the h1 to the feed-menu name, a subsequent reload that errors/empties keeps the old title (filed as issue #2). The header then misrepresents the current state.
  - **Why it matters:** Directly undercuts the "honest about state" principle.
  - **Fix:** Reset the h1 to "Feed Menu" in `loadFeedMenu`'s reset block (one line).
  - **Suggested command:** /impeccable harden

- **[P2] No concept-level help for first-timers.** The onboarding page teaches *pinning*, not *what feeds are or why to use one*. There's no help affordance in the popup itself.
  - **Why it matters:** The mainstream user never gets the "why."
  - **Fix:** A short explainer line in the empty state and/or onboarding; optionally a subtle help link.
  - **Suggested command:** /impeccable onboard

- **[P3] Dead-end if no feed reader is installed.** Clicking a feed flips to "Opening in Reader…" for 2s via `feed://`; if no handler exists, nothing happens and the badge just disappears.
  - **Why it matters:** Silent no-op is the worst failure mode for a newcomer.
  - **Fix:** Detect/handle the no-handler case, or offer "copy feed URL" as a fallback.
  - **Suggested command:** /impeccable harden

## Persona Red Flags

**Jordan (First-Timer — the PRODUCT.md primary persona):** Reads "No feed menu found… Checked /.well-known/feed-menu.json" and learns nothing actionable. Never told what a feed *is* or what "Opening in Reader…" means. Most likely to bounce at the empty state.

**Sam (Accessibility):** Strong overall — keyboard submenu, focus rings, AA contrast, `role="status"`/`role="alert"`, error uses icon + color (not color alone). Red flag: the transient "Opening in Reader…" badge isn't in a live region, so a screen-reader user gets no spoken confirmation that the handoff started.

**Riley (Stress Tester):** Finds the stale-title bug (#2); finds the silent dead-end when no feed reader is installed. Long titles/descriptions are handled (ellipsis + `break-word`) — good. Refresh mid-state is moot (popup reloads fresh on open).

## Minor Observations

- "Server error: 500" / "malformed" messages are accurate but slightly technical; fine as secondary detail, not as the whole message.
- The discovery toggle's inline help is well done — explains the permission cost honestly.
- Empty vs. error are well separated now; keep that line bright.

## Questions to Consider

- What would the empty state say if it assumed the user had never heard the word "feed"?
- Should a transient failure offer "Try again" right where the error is, instead of sending the eye to a header icon?
- Is "Opening in Reader…" a promise you can keep for a user with no reader installed?
